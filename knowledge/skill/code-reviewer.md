# code-reviewer

## Summary
对 Unity + Lua 项目的待提交变更进行风险导向审查，输出可执行的审查报告与结论

## Content
## 审查目标

- 优先识别会导致线上事故或用户损失的问题
- 输出可执行、可定位、可修复的审查意见
- 给出明确结论：`PASS`、`PASS_WITH_RISKS` 或 `BLOCK`

## 审查范围

1. 默认范围：`git diff --name-only HEAD`
2. 仅审查 `Assets`、`Assets.Lua`、`Packages`、`Players`、`ProjectSettings` 目录文件
3. 仅审查文本文件，跳过二进制与临时文件
4. 若存在已暂存内容，结合 `git diff --cached` 一并审查

## 审查流程

1. 收集变更文件与核心 diff
2. **读取 CLAUDE.md 和项目文档** 以获取项目编码标准
3. 按「通用检查清单」逐项审查
4. 对 Lua 与 Unity C# 变更追加「项目专项检查」
5. 按统一分级标准标注问题严重度
6. 输出结构化报告与最终结论

## 通用检查清单

### 1. 正确性

- [ ]  业务逻辑是否与需求一致，是否出现条件分支遗漏
- [ ]  边界值覆盖：空值、零值、负值、上限、下限、异常输入
- [ ]  数组越界、空引用、除零、整数溢出、精度损失
- [ ]  递归/循环是否有明确退出条件，是否可能死循环

### 2. 安全性

- [ ]  输入是否经过校验与过滤，是否存在注入风险
- [ ]  路径、文件、URL、序列化是否存在越权或路径穿越风险
- [ ]  敏感信息（token、密钥、用户隐私）是否被日志或代码泄露
- [ ]  依赖变更是否引入已知高危漏洞

### 3. 稳定性

- [ ]  外部调用（网络、磁盘、IO、SDK）是否有失败兜底
- [ ]  异常是否被正确捕获、上报和可观测（日志含关键信息）
- [ ]  资源是否成对释放（连接、句柄、对象池、临时资源）

### 4. 性能

- [ ]  热路径（Update/循环/高频回调）是否引入不必要分配或重计算
- [ ]  是否存在 N+1 查询、循环内远程调用、重复 I/O
- [ ]  算法复杂度是否与数据规模匹配，是否可缓存或批处理

### 5. 可维护性

- [ ]  命名是否准确清晰，模块职责是否单一
- [ ]  是否有明显重复逻辑
- [ ]  复杂逻辑是否有必要注释或拆分
- [ ]  是否遵循项目约定（`.editorconfig`、目录结构、既有模式）

### 6. 测试覆盖

- [ ]  新增或变更行为是否有对应测试
- [ ]  是否覆盖关键分支与失败分支
- [ ]  是否说明验证步骤和预期结果

## 项目专项检查

### Lua 专项


| 检查项     | 说明                                                                                      |
| ---------- | ----------------------------------------------------------------------------------------- |
| 全局变量   | 避免使用全局变量，减少作用域污染                                                          |
| 循环引用   | 检查表之间的循环引用导致的内存泄漏；对象池归还时须断开互相引用                            |
| 热路径分配 | 热路径避免临时 table/闭包；若复用表须 `table.clear` 或等价全键清除，且避免引用外泄给 callee |
| GC 压力    | 避免大字符串拼接、频繁 table 操作；热路径缓存 `CS.*`、少用 `transform.position` 整表读写  |
| 对象池     | Acquire 后强制重置全部字段；池有上限；警惕循环引用；评估创建开销是否真有必要                |
| 并发安全   | 网络消息处理考虑并发顺序、重入、防抖与上限；复用表不在嵌套调用间共享                      |
| 生命周期   | 协程/定时器是否存在泄漏（对象销毁后仍回调）                                               |
| 函数本地化 | 常用函数在文件开头用 `local assert = assert` 形式本地化                                   |
| 表引用     | 复用表/shared args 不被 callee 长期持有；减少表中成员是另一表的强引用                     |
| 复杂度控制 | 高级特性尽量不用，代码保持简单；无热路径证据的池化/复用仅作 LOW 建议                      |

### Lua 复用表与对象池（审查专用）

与 `skill/gc.md` 的分工：**gc.md** 描述优化写法与热点模式；本节定义审查风险与严重度。**正确性优先于 GC**；缺少复用/池化不得单独构成 BLOCK。

**热路径：** Update / FixedUpdate 等价回调、每帧 timer、高频网络 handler、紧凑循环。

#### 通用热路径 GC（审查视角）

| 模式 | 典型问题 | 建议严重度 |
| ---- | -------- | ---------- |
| 每帧 `local t = {}`、`table.pack`、函数返回新表 | 可避免的分配压力 | MEDIUM（建议） |
| 循环内 `..` / `string.format` / `tostring` | 字符串 churn | MEDIUM |
| 循环或每帧新建匿名函数/闭包 | 闭包分配 | MEDIUM |
| 热路径未缓存 `CS.UnityEngine.*` | 查找与临时 userdata | MEDIUM |
| 频繁读写 `transform.position` / rotation | struct → Lua table | 核心路径 MEDIUM–HIGH |
| `pairs` / `ipairs` 循环体内再建临时表 | 每次迭代分配 | MEDIUM |
| 由上述模式可推断的泄漏或帧抖动 | GC 债务 | HIGH |

冷路径 `local t = {}` 且无热路径证据 → 至多 LOW，不得升为 HIGH。

#### 复用表审查清单

变更涉及 module 级 `_temp*`、共享 args 表、或「清空后重用」模式时**必须逐项检查**：

| 审查点   | 通过标准                                                                | 典型问题                                   | 建议严重度      |
| -------- | ----------------------------------------------------------------------- | ------------------------------------------ | --------------- |
| 完全清空 | 重用前 `table.clear(t)`（或等价全键清除）；数组与 hash 部分均处理       | 只改部分字段，旧 key 残留导致逻辑错误      | **HIGH**        |
| 嵌套子表 | 复用前对每个嵌套 table 字段单独 `table.clear` 或重建；父表 `table.clear` 不递归 | 父表清空后子表仍含上次数据                 | **HIGH**        |
| 引用影响 | module 级 `_temp` 不外泄给长生命周期对象/全局/callback                  | 调用方缓存表引用，下一帧数据被覆盖         | **HIGH**        |
| 并发/重入 | 同一 `_temp` 不在嵌套调用间共享                                         | 重入时字段互相覆盖                         | **HIGH**        |
| 元表干扰 | 复用表无会阻止清空的 `__newindex`/`__index` 副作用；weak table 语义明确 | `table.clear` 后行为与预期不符             | **MEDIUM–HIGH** |
| 必要性   | 热路径有分配证据或 profiler / `lua_gc_analyze` 依据                     | 冷路径强行 module 级复用，可读性下降无收益 | **LOW**         |

#### 对象池审查清单

变更涉及 `*Pool*`、`Acquire`/`Release`、或 acquire-return 模式时**必须逐项检查**：

| 审查点   | 通过标准                                              | 典型问题                         | 建议严重度          |
| -------- | ----------------------------------------------------- | -------------------------------- | ------------------- |
| 强制重置 | `Acquire` 后重置全部字段；嵌套子表须单独清空或重建；归还前断开对外引用 | 上次使用或嵌套表内容泄漏到下次 acquire     | **HIGH**            |
| 池大小   | `maxSize` 或丢弃策略；满池时 release 销毁而非无限增长 | 无上限 `table.insert(pool, obj)` | **MEDIUM**          |
| 循环引用 | 池对象间、池↔外部无强引用环；`Release` 清空互相引用   | 池 + 闭包/upvalue 导致无法 GC    | **HIGH**            |
| 归还时机 | 确认无其他活跃引用再入池                              | 入池后仍被外部修改               | **HIGH**            |
| 创建开销 | 有热路径频率或 profiling 依据                         | 低频对象强行池化，复杂度 > 收益  | **LOW**（建议简化） |

#### Lua GC 结论边界

- **不得**仅因「未做复用/池化」给出 BLOCK。
- **应当**对错误复用/池化（脏数据、引用泄漏、未重置、无上限池、循环引用）给 **HIGH** 或 **BLOCK**。
- 过早优化标 **LOW**，整体仍可 **PASS** / **PASS_WITH_RISKS**。

### Unity C# 专项

#### 热路径优化


| 检查项          | 说明                                                                   |
| --------------- | ---------------------------------------------------------------------- |
| Update 避免操作 | 禁止在 Update/FixedUpdate/LateUpdate 中：分配、反射、Find/GetComponent |
| 昂贵计算外移    | 将复杂计算移出 Update，使用协程或事件系统                              |
| 缓存引用        | 在 Start/Awake 中缓存组件引用，避免运行时 Find                         |

#### 内存与资源


| 检查项     | 说明                                             |
| ---------- | ------------------------------------------------ |
| 值类型优先 | 性能敏感场景优先使用 struct 而非 class           |
| 对象池     | 频繁创建销毁的对象使用对象池，减少 GC 压力       |
| 资源释放   | IDisposable 对象使用`using` 语句，确保资源释放   |
| 异步加载   | 资源加载优先异步，遵循 Load/Release 引用计数规则 |

#### 代码规范


| 检查项        | 说明                                      |
| ------------- | ----------------------------------------- |
| readonly 字段 | 不需修改的字段使用 readonly 修饰符        |
| 类型转换      | 使用`is`/`as` 而非强制类型转换            |
| 继承层次      | 减少 MonoBehaviour 继承层次，优先使用组合 |
| 字符串拼接    | 避免循环中用`+` 拼接，使用 StringBuilder  |
| LINQ 慎用     | 性能敏感场景谨慎使用 LINQ，避免不必要分配 |

#### 数据与对象管理


| 检查项           | 说明                                             |
| ---------------- | ------------------------------------------------ |
| ScriptableObject | 游戏数据使用 ScriptableObject 管理，减少内存使用 |
| List vs 数组     | 动态大小优先使用`List<T>` 而非数组               |
| 协程异步         | 异步操作使用协程，提高可读性和可维护性           |

## 问题分级标准

### CRITICAL（阻断）

满足任一条件即判定：

- 可导致安全漏洞、数据泄露、权限绕过
- 可导致崩溃、数据损坏、核心流程不可用
- 高概率造成严重性能故障（卡死、雪崩、资源耗尽）

### HIGH（阻断）

- 明确功能错误或高概率线上故障
- 核心场景缺少必要异常处理或边界处理
- 明显性能回退且位于核心路径

### MEDIUM（建议修复）

- 可维护性问题明显，后续改动风险高
- 测试覆盖不足，但当前有替代验证证据
- 非核心路径性能可优化项

### LOW（可跟进）

- 风格一致性、文档、命名微调
- 不影响行为正确性的轻量改进建议

## 审查结论规则


| 结论              | 条件                            |
| ----------------- | ------------------------------- |
| `BLOCK`           | 存在 CRITICAL 或 HIGH           |
| `PASS_WITH_RISKS` | 无 CRITICAL/HIGH，但存在 MEDIUM |
| `PASS`            | 仅 LOW 或无问题                 |

## 输出报告模板

### Findings（按严重度降序）

```
1. [CRITICAL] 文件路径:行号
   - 问题：说明风险
   - 影响：影响范围或故障模式
   - 建议：最小可行修复方案

2. [HIGH] 文件路径:行号
   - ...
```

### Open Questions（可选）

需要作者确认的上下文或设计意图。

### Final Verdict

```
BLOCK / PASS_WITH_RISKS / PASS
一句话说明结论依据。
```

## 执行原则

- 只基于证据下结论，不做无依据猜测
- 优先报高价值问题，避免泛泛而谈
- 给出可落地修复建议，不只指出问题
- 绝不批准存在已确认安全漏洞的代码
- **正确性优先于 GC**；无热路径证据的池化/复用仅作 LOW 建议，不得单独 BLOCK

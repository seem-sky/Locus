## Common Unity Implementation Pitfalls

* Do not guess the default execution order between scripts: the default order of `Awake`, `OnEnable`, and `Update` across different scripts is arbitrary; cross-object initialization dependencies should be explicitly designed, and Script Execution Order should be configured in Project Settings when necessary.
* Separate responsibilities among `Update` / `FixedUpdate` / `LateUpdate`: physics calculations, force application, and velocity changes should usually go in `FixedUpdate`; per-frame state updates should go in `Update`; camera follow logic and logic that needs to override animation results should go in `LateUpdate`.
* Confirm scene load callback timing independently: `SceneManager.sceneLoaded` happens after `OnEnable` and before `Start`; when restoring references across scenes, registering systems, or doing secondary initialization, choose the correct entry point.
* Unity physics behavior is affected by the collision matrix in project settings and the Layer settings of each GameObject. When you need to write physics-related logic (`Collider`, `Rigidbody`, `Trigger`, `Raycast`, `Overlap`, etc.), be sure to confirm the current physics settings of the project.
* Trigger / Collision callbacks have prerequisites: for Trigger to work, at least one object in the interacting pair must have a `Rigidbody`; when callbacks do not fire, check both sides' `Collider`, `Is Trigger`, `Rigidbody`, `Layer`, and the collision matrix.
* 2D and 3D physics systems are two separate systems: `Collider` / `Rigidbody` / `Physics` and `Collider2D` / `Rigidbody2D` / `Physics2D` cannot be mixed; when troubleshooting collisions, raycasts, triggers, or project settings, first confirm that you are in the same system.
* Check raycast and query logic thoroughly: confirm `LayerMask`, `queryTriggerInteraction`, the global `Physics.queriesHitTriggers`, and note that a ray whose origin starts inside a Collider will not register a hit.
* Do not assume Unity serialization follows ordinary C# rules: Unity serializes fields rather than properties by default; private fields require `[SerializeField]`; `Dictionary<>` is not supported by default serialization; `JsonUtility` cannot directly handle top-level arrays or `List<T>` and requires wrapping them in an object.
* Enter Play Mode configuration changes debugging conclusions: the project may have Domain Reload or Scene Reload disabled; when Domain Reload is disabled, static fields and static event handlers will not reset automatically, and many “only breaks on the second run” issues in the editor come from this.
* The null semantics of Unity Objects are different from ordinary C#: once the underlying object of a `UnityEngine.Object` has been destroyed, `obj == null` may be true while `ReferenceEquals(obj, null)` is still false; `?.` and `??` also do not apply to Unity Objects that may enter a detached state.
* Avoid lookups and allocations in hot paths: do not repeatedly use `Find`, `FindObjectOfType`, `GetComponent`, LINQ, or string concatenation inside `Update` / `FixedUpdate`; prefer caching references during initialization, reusing collections, and using non-alloc API variants.
* Distinguish object destruction APIs between runtime and editor: prefer `Destroy` at runtime; `DestroyImmediate` is mainly for Edit Mode.
* Be explicit about thread boundaries: most `UnityEngine` / `UnityEditor` APIs can only be called on the main thread; after getting results from async tasks, background threads, or callback threads, switch back to the main thread before accessing `GameObject`, `Transform`, or `Component`.
* Do not assume changing a field’s default value in a C# script also updates the actual values already serialized in scene GameObjects or Prefabs; verify their actual serialized values directly.
* When modifying an object's position or size in a scene, keep in mind that a GameObject's Transform can only be edited directly in local space. Its final world-space position and size are determined by its parent and ancestor Transforms. The `unity_yaml_read` utility returns a GameObject's world-space position and size when reading a target object.

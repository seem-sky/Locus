# Plugin Publishing

Workflow for publishing an exported Locus plugin to GitHub and the plugin registry. Complete the audit, id approval, and `plugin_export` steps in `SKILL.md` first; every path below starts from an approved plugin id, version, and exported plugin root.

## Shared rules

- The official Locus plugin registry is always `r1n7aro/locus-plugin-registry`. Use it directly when the user asks for the official registry, asks for a registry PR, or does not name a registry repository. Never infer a different registry from plugin id, plugin repository name, GitHub login, package namespace, or organization name. Ask for the registry repository only when the user explicitly requests a custom registry or names more than one candidate.
- Resolve only the base branch: the registry default branch for real publishing, or a user-confirmed branch such as `test` for validation.
- The registry source entry for a plugin is `entries/v1/plugins/<bucket>/<plugin-id>.json`, where bucket is the first two hex chars of the SHA-256 of the plugin id.
- `public/v1/**` (`manifest.json`, `shards/<bucket>.json`, `plugins/<bucket>/<plugin-id>.json`, `search/summaries.json`) is generated by registry CI and owned by the registry repository. Never modify it in a PR. CI validates source entries on PRs; pushes, scheduled runs, and manual dispatches rebuild `public/v1` from `entries/v1` and GitHub release assets.
- GitHub work uses `bash` with `gh`. Confirm authentication before any GitHub step; if it is missing, ask the user to run `gh auth login -h github.com -s repo,read:org`.
- The confirmed plugin id is stable release identity. Future registry entries and archive names keep that id until the user explicitly chooses a replacement id and handles migration. The registry repository owner and GitHub login never decide the plugin id.

## Choose the publish path

- GitHub repository and release → section A.
- First registry registration, or a change to registry metadata or download source rules → section A, then section B.
- Version-only update of an already registered plugin → section C. It needs no registry PR.

## A. GitHub repository and release

- The plugin repository root must be the installable plugin source tree: a root `locus.plugin.json`, component directories such as `views/`, `skills/`, `agents/`, and `rules/`, plus README/LICENSE files. Do not create a repository that only contains README/LICENSE and a release asset, and do not hide the plugin under `release/`, `dist/`, or another nested folder — GitHub repo installs clone the repository and locate `locus.plugin.json` at the root.
- Build the repository tree from the same exported plugin root used for the release asset. The repository root, the GitHub source archive, and the release zip must all contain a root `locus.plugin.json` with the same plugin id and version.
- Check repository availability with `gh repo view <repo-owner>/<repo-name>`. If it already exists, ask before reusing it for release assets.
- After publishing or replacing a release asset, list the latest release assets and resolve the intended `downloadSource` rule. If `assetPattern` is used, it must match exactly one asset; if it matches zero or multiple assets, fix the release assets or selector before opening a registry PR and before declaring the release complete.
- Verify the final release archive: download or inspect the exact asset, compute SHA-256 and byte size, and read its root `locus.plugin.json` to confirm id and version. Then verify the repository source separately — install from the GitHub repo or inspect its default-branch archive. Both paths must resolve to the same id, version, and component tree.

## B. Registry registration PR

Write the source entry:

- A new plugin adds exactly one file: `entries/v1/plugins/<bucket>/<plugin-id>.json`. An existing plugin updates that same file only when registry metadata or download source rules change.
- Include the user-facing metadata: `author`, `repo`, `license`, `summary`, `description`, `tags`, compatibility, dependency metadata, and `stats`. Use current registry fields only; do not use legacy or npm-style fields such as `schema`, `version`, `homepage`, `repository`, or `components`.
- Include standard stats definitions for GitHub-hosted plugins: `githubStars` with label `Stars` and `releaseDownloads` with label `Release downloads`. Registry CI refreshes the stat values.
- Store `downloadSource`, never generated `download`, `latestVersion`, or release SHA fields; CI resolves `latestVersion`, `download.url`, `download.sha256`, `download.sizeBytes`, `updatedAt`, and `downloadSource.version` into `public/v1`. Use `downloadSource.type: "latestRelease"` for normal GitHub release publishing. Omit `asset` when each release has exactly one plugin zip. Use `assetPattern` such as `locus-workspace-*.zip` when asset names include the version, and never set `asset` to a versioned filename. A release with multiple zip assets must set `asset` or `assetPattern` so CI fails instead of guessing.
- After the user confirms the plugin id, check registry duplicates on the target base branch: compute the bucket and read the entry path. If the id exists, ask whether this is a new version of that plugin, a replacement or fork under a new id, or a stop.
- Before opening the PR, confirm the registry metadata with the `sheet` tool in one form: target repository, plugin id, author, release source rule, description source, license, compatibility, dependency metadata, plugin purpose, usage instructions, and stats definitions. Apply user edits from the confirmed sheet to the source entry. After a change request, revise and present an updated sheet before continuing.

Open the PR from a fork:

- The official registry is the PR base repository; all writable branches for registry changes belong to the user's fork.
- Run one GitHub preflight: `gh auth status -h github.com`, `gh api user --jq .login`, and `gh repo view r1n7aro/locus-plugin-registry --json nameWithOwner,defaultBranchRef`.
- Ensure a fork with `gh repo fork r1n7aro/locus-plugin-registry --clone=false --remote=false`, then verify `<viewer>/locus-plugin-registry` exists.
- Create a unique fork branch from the upstream base SHA, such as `register/<plugin-id>/<version>-<timestamp>`.
- Prefer `gh api repos/<fork-owner>/locus-plugin-registry/contents/<path> -X PUT --input -` for registry file writes; it avoids cloning and stays stable in constrained networks. Creating a new entry uses `message`, `branch`, and `content`. Updating an existing entry must first read the file on the target branch and include its current blob `sha` in the PUT payload.
- Write the PR body to a Markdown file and pass it with `--body-file <body.md>`. Write literal Markdown: no escaped `\n` strings and no shell quoting that treats Markdown backticks as escape characters. In PowerShell, prefer a single-quoted here-string (`@' ... '@`).
- Include these PR body sections:
  - `## Summary`: registration target, plugin id, version, and release link.
  - `## What This Plugin Does`: user-facing purpose and the included Skill/View/Rule components.
  - `## How To Use`: install path plus the natural way to invoke included Skills or open included Views.
  - `## Registry Metadata`: plugin repo, author, stats definitions, download source rule, current release link, bucket, compatibility, dependencies, license, description source, and the single source entry path.
  - `## Validation`: export, local install and ownership transfer, plugin repository source installability, release archive inspection, source entry update, registry CI resolves generated version/download metadata, and `public/v1/**` left unchanged.
- Create the PR with `gh pr create --repo r1n7aro/locus-plugin-registry --base <base-branch> --head <viewer>:<fork-branch> --title "Register <plugin-id> <version>" --body-file <body.md>`.
- Verify the PR with `gh pr view --repo r1n7aro/locus-plugin-registry <number-or-url> --json url,state,baseRefName,headRefName,headRepositoryOwner`, and confirm the changed files contain only the single entry file. On PR checks, expect registry validation to pass and the generated-index build to be skipped.

After merge:

- Watch the base-branch registry workflow. The follow-up generated commit should touch only `public/v1/manifest.json`, `public/v1/search/summaries.json`, `public/v1/shards/<bucket>.json`, and `public/v1/plugins/<bucket>/<plugin-id>.json`.
- Validate the generated index with the GitHub contents API as the immediate source of truth: source entry, public detail entry, bucket shard, manifest `entryCount` and `availableBuckets`, and `search/summaries.json` must all agree on id, version, URL, SHA-256, size, compatibility, and dependency metadata.
- GitHub raw branch URLs can lag briefly after a generated commit. If raw and contents API disagree right after merge, wait and retry before treating it as a registry index failure.
- Download the public entry archive URL and verify SHA-256, byte size, and root `locus.plugin.json` id/version against the generated public entry. Also install from the plugin repository or download its source archive; it must import as the same plugin id and version as the release asset.

## C. Version-only release

- Update and commit the plugin repository root source tree, publish the GitHub release asset from that committed source, and let registry CI refresh `public/v1`.
- Do not create or update `entries/v1` and do not open a registry PR unless metadata, compatibility, dependency metadata, description, icon, tags, repo, license, or download source rules change.
- Apply the release-asset checks from section A: selector resolution, archive verification, and repo/release manifest agreement.
- Verify by triggering or waiting for registry CI, then inspecting only the `public/v1` generated outputs.
